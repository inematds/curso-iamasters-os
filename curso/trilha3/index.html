👥 Multi-cliente

Uma instalação, N clientes. Quando ativar, como criar, 4 templates verticais e separação segura de dados entre clientes.

O que é:

Multi-cliente é a capacidade do OS de servir N clientes a partir de UMA instalação, mantendo voice, ICP e positioning separados por cliente em clients/<nome>/.

Por que aprender:

Para decidires bem o teu setup. Ativar multi-cliente sem o precisar adiciona fricção; não o ativar quando precisas leva a misturar contextos e gerar copy errada.

Conceitos-chave:

Freelance/agência com 2+ clientes → ativa · single-business ou marca pessoal sozinha → raiz · "verticais do mesmo produto" não são multi-cliente.

O que é:

Script bash scripts/add-client.sh <nome> <vertical> que cria clients/<nome>/ com brand-context, context, projects, CLAUDE.md de overrides e substitui placeholders {{CLIENT_NAME}}.

Por que aprender:

Para arrancares um cliente em <1 min sem esqueceres ficheiros. O script valida o nome (kebab-case), recusa duplicados e indica os próximos passos.

Conceitos-chave:

Validação kebab-case · modo `vacio` sem template · placeholder replacement automático · CLAUDE.md por cliente.

O que é:

4 pastas em clients/_templates/ com voice, positioning e ICP pré-preenchidos para o vertical: freelance-ia (operador solo) · agencia-marketing (2-10) · formador-online (cursos) · consultoria-b2b (high-ticket).

Por que aprender:

Para escolheres o template certo — economizas 30-60 min de scaffolding por cliente. Cada template trá tom, vocabulário proibido e padrões já adequados.

Conceitos-chave:

Vertical como ponto de partida (não destino) · placeholders {{...}} para preencher · template `vacio` se nenhum encaixar.

O que é:

Quando fazes cd clients/acme && claude, o Claude lê o CLAUDE.md raiz (instruções gerais) + clients/acme/CLAUDE.md (overrides) + brand-context e context do cliente. Skills da raiz continuam disponíveis.

Por que aprender:

Para colocares regras específicas por cliente no sítio certo (override) e regras gerais na raiz. Saber a ordem evita conflitos.

Conceitos-chave:

Skills custom por cliente em clients/<X>/.claude/skills/ · Sinapsis (operator-state) é global · brand-context é por cliente.

O que é:

O ritual: cd clients/acme && claude → trabalha → /wrap-up → exit → cd ../widget-shop && claude → repetir. Wrap-up entre clientes guarda daily summary específico de cada um.

Por que aprender:

Sem este ritual misturas contextos. Wrap-up ao mudar cliente é regra de ouro — separa learnings, decisions-log e summaries.

Conceitos-chave:

"Sempre cd antes de claude" · 2 terminais para sessões simultâneas · nunca trabalhar cliente desde a raiz.

O que é:

Por design, as skills NÃO partilham dados entre clientes. clients/<nome>/ está no .gitignore por defeito. Update.sh nunca toca em ficheiros de clientes.

Por que aprender:

Para não acidentalmente usares case study de Acme numa proposta a Widget. Se queres reutilizar info, geras "case study anonimizado" como output do cliente A e referencias.

Conceitos-chave:

.gitignore por defeito · update.sh respeita user-data · anonimização consciente · backup separado para clientes críticos.

⚙️ Operação diária

Os rituais que mantêm o OS vivo: start-here, wrap-up, decisions-log, deep-dive, doctor e update. O que fazer cada dia.

O que é:

Skill que lê operator-state, daily summary de ontem, learnings, projetos abertos e proposta para hoje. Saúda com: "Ontem deixaste X. Continuas com Y ou mudamos?".

Por que aprender:

Sem ritual de início, perdes 5-10 min cada sessão a relembrar onde ias. Com start-here, em 30s estás operacional.

Conceitos-chave:

Auto-deteção needsOnboarding · cumprimento contextual · deteção de skills pendentes · lembrete deep-dive se aplicável.

O que é:

Ritual de fecho em 9 passos: recap da sessão, sync skills-catalog, atualiza CLAUDE.md registry, append learnings, gera daily summary, deteta projetos para arquivar, propõe commit (espera aprovação), trigger Sinapsis EOD, despedida.

Por que aprender:

É o que faz amanhã haver memória de hoje. Sem wrap-up, daily summary não se gera, o /start-here de amanhã não tem nada para ler.

Conceitos-chave:

Daily summary em synapsis/daily-summaries/ · "For tomorrow" · conventional commits · NÃO push automático.

O que é:

Skill que regista decisões importantes em formato canónico (Decisão · Raciocínio · Contexto · Quando reconsiderar). Append-only — nunca edita entradas. Inspirada em claude-code-second-brain de Luis Pitik.

Por que aprender:

Para o sistema (e tu) não se contradizer entre sessões. Para teres track record honesto. Para reavaliares decisões com critério, não com memória vaga.

Conceitos-chave:

Decisão qualifica se: não trivial + reversibilidade limitada + influencia futuras conversas · inversão = NOVA entrada · 3 entradas/sessão máximo.

O que é:

Segunda fase do onboarding após o wizard inicial. Entrevista conversacional adaptativa em 22-25 dimensões (pessoa, negócio, equipa, foco). 25-30 min. Idempotente — retoma onde ficou.

Por que aprender:

É o que converte outputs "decentes" em outputs que parecem teus. Sem deep-dive, o sistema sabe o teu nome e stack — não conhece os teus motivadores, fricções, decisões pendentes.

Conceitos-chave:

Branching condicional (sozinho → salta bloco equipa) · 2 níveis máximo de aprofundamento · escreve em context/me, work, team, soul, current-priorities, goals.

O que é:

Health check v0.6 com validação profunda: JSON parseable, hooks executáveis, conteúdo >100 chars em ficheiros críticos. Cruza com install-state.json para detetar drift. Auto-fix para problemas reversíveis.

Por que aprender:

Quando algo falha sem dares conta — apagaste ficheiro, mudaste config — /doctor diz-te o que está partido em 30s e propõe fixes seguros.

Conceitos-chave:

🟢🟡🔴 report · STATE DRIFT detection · auto-fix com confirmação "sí, revertir" para drift · não recolhe dados pessoais.

O que é:

Script bash scripts/update.sh que sincroniza o repo com upstream: skills atualizadas, templates de clientes refrescados, vendor/sinapsis/ refeito. NUNCA toca em clients/<nome>/, brand-context/, context/ ou projects/ teus.

Por que aprender:

Para não te desatualizares (skills novas, fixes) sem medo de perder o teu. A regra de ouro: separação user-data vs system-files.

Conceitos-chave:

Backup automático antes · conflict resolution interativo · valida com /doctor após update · re-run install se mudou state schema.

🧩 Estender o sistema

install-skill, install-mcp, catálogos curados, plugins Anthropic, e como criar a tua própria skill e contribuir ao catálogo.

O que é:

Comando que descarrega skill de GitHub para temporário, valida (SKILL.md, YAML frontmatter, description 50-500 chars com verbo, sem código perigoso), mostra resultado, instala e testa ativação.

Por que aprender:

Para esticares o catálogo conforme precisas, sem inflar com lixo. Validação bloqueia skills mal feitas. Backup automático se houver conflito.

Conceitos-chave:

Modo atalho (`/install-skill cognito` para skills em _optional/) · validate-skill.sh · conflict resolution · teste de ativação.

O que é:

Comando que adiciona MCP server ao .mcp.json do projeto desde lista curada (top 5: context7, github, supabase, notion, firecrawl) ou URL custom. Configura env vars e permissions.

Por que aprender:

MCPs estendem o Claude para o teu stack real (BD, CRM, repos). Saber escolher os 5-7 certos vs encher de tudo poupa tokens.

Conceitos-chave:

.mcp.json estrutura · variáveis em .env · permissions extra · READ-ONLY antes de write · 5-7 MCPs ativos máximo (token budget).

O que é:

Dois docs em docs/: skills-recommended.md (Camada 2 — 35+ skills opcionais: copy, CRO, SEO, analytics, etc.) e mcps-curated.md (top 5 + úteis específicos + MCPs a evitar).

Por que aprender:

Para descobrires o que existe sem caçar pela internet. Cada entrada tem caso de uso, dificuldade, riscos. Curado = filtrado por uso real ≥2 semanas.

Conceitos-chave:

"Max 30 skills, não 1000" · revisão trimestral · pattern para a tua lista por cliente · alternativa lean: claude-code-second-brain do Luis Pitik.

O que é:

Plugin /plugin install anthropic-skills que adiciona 4 skills oficiais Anthropic: docx (Word), xlsx (Excel/CSV), pptx (PowerPoint), pdf (ler, combinar, dividir, marcar).

Por que aprender:

Para qualquer operador que receba ou entregue Office. Recomendação: instalar SEMPRE — é a maioria dos clientes.

Conceitos-chave:

Marketplace Anthropic vs install-skill (GitHub) · skills oficiais com manutenção · acesso a binários office sem libs externas.

O que é:

Skill que cria skills novas. 9 passos: requisitos → valida nome/descrição → gera estrutura (SKILL.md + references/ + scripts/ opcional) → escreve template → regista no catálogo → teste mínimo de ativação.

Por que aprender:

Quando detetas um padrão repetido em 3+ sessões, é hora de o "graduar" para skill. Sem meta-skill-creator, criar skill manual demora 30-60 min e parte-se algo.

Conceitos-chave:

3 testes de descrição (ativação, comprimento 50-500, unicidade) · progressive disclosure (references/) · output-verifier gate · teste de ativação obrigatório.

O que é:

Processo para propor skill nova ao catálogo público iAmasters: validada ≥2 semanas em produção, serve ≥3 avatares, output útil sem contexto, não depende de info confidencial. PR em iamasters-academy/iamasters-os.

Por que aprender:

Para devolveres ao ecossistema o que aprendeste. As tuas skills validadas podem servir 100 outros operadores. Revisão trimestral.

Conceitos-chave:

Critérios objetivos · 1-2 exemplos de uso real obrigatórios · categoria sugerida · porque encaixa na Camada 2 · maintainer (Angel) revê.
