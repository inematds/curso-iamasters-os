▶️ /start-here — ritual de início

Cada sessão começa com /start-here (ou automaticamente no primeiro turno). Em 30s o sistema lê tudo o que precisas saber para arrancar e propõe a tarefa do dia.

💡 O que /start-here lê

  • •~/.claude/skills/_operator-state.json — perfil do operador, deteta needsOnboarding
  • •context/me.md — identidade (deriva para wizard se vazio)
  • •synapsis/daily-summaries/<ONTEM>.md — "For tomorrow" da sessão anterior
  • •context/learnings.md — última lição adicionada
  • •projects/briefs/*/brief.md com status: active — projetos abertos
  • •context/decisions-log.md — últimas 5 decisões (para coerência)

3 modos de cumprimento

# Modo 1 · Continuidade (há daily summary de ontem)
"Olá Nei. Ontem fechaste com: blog post Stripe billing (status: pending review).
 Para hoje propunhas: passá-lo pelo output-verifier e publicar.
 Continuas com isso, ou mudamos?"

# Modo 2 · Projeto ativo sem summary recente
"Olá Nei. Tens o projeto **landing-iamasters** aberto na fase copy.
 Continuas com ele ou vamos a outra coisa?"

# Modo 3 · Sem atividade recente
"Olá Nei. Em que te ajudo hoje?
 [1] Criar conteúdo (skills marketing-*)
 [2] Trabalhar com um cliente (cd clients/<x>)
 [3] Análise estratégica
 [4] Tarefa livre"

💡 Lembrete de deep-dive

Se completaste o wizard inicial mas não o deep-dive, o cumprimento termina com PS sutil ("ainda não completaste o deep-dive — quando tiveres 25 min, executa /deep-dive"). Lembra todos os dias até completares.

⏹️ /wrap-up — fecho + commit + daily summary

/wrap-up é o ritual de fecho. Sem wrap-up, amanhã não há memória de hoje. 9 passos automáticos, ~1 minuto.

1

Recap mental + sync skills

Resume o que se completou, o que ficou a meio, o que se aprendeu. Lê .skills-pending.json e sincroniza skills-catalog.json se houve skills novas.

2

Update CLAUDE.md skills registry

Localiza bloco entre <!-- skills-registry-start --> e end, regenera a tabela. Sem isto, o registry desincroniza-se do filesystem.

3

Append learnings (se aplicável)

Se uma skill falhou, append em context/learnings.md com data + razão + fix. Próxima vez essa skill carrega o aprendido.

4

Gerar daily summary

Cria/atualiza synapsis/daily-summaries/<TODAY>.md com Sessions, Done, Pending, Learnings, Decisions + "For tomorrow" + "Quick resume". Se houver sessões prévias hoje, faz append.

5

Commit Git (com aprovação)

Mostra git status, propõe mensagem (conventional, em inglês: feat(skills): add X, chore(wrap-up): EOD <data>). Espera "sim" explícito. Nunca push automático.

⚠️ NÃO se invoca automaticamente

Fechar Claude Code com Ctrl+C não dispara wrap-up. Tens de pedi-lo explicitamente ou usar /wrap-up. Se esqueces, perdes o daily summary daquela sessão.

3

📓 decisions-log — diário append-only

O decisions-log é o que impede o Claude (e tu) de se contradizer entre sessões. Formato canónico com 4 campos. Append-only — entradas nunca se editam. Padrão inspirado em claude-code-second-brain de Luis Pitik.

Formato canónico

## YYYY-MM-DD · <título curto, ≤60 chars>

**Decisão**: <1 frase clara com o quê>

**Raciocínio**: <2-4 frases com o porquê — inclui os dados ou
sinais que levaram à decisão, não só "porque sim">

**Contexto**: <1-2 frases com a situação mais ampla que explica
porque esta decisão fazia sentido neste momento. Crucial para
reavaliar amanhã.>

**Quando reconsiderar**: <critério concreto que faria rever a
decisão — evento, métrica, data. NÃO "se correr mal".>

---

📊 Uma decisão qualifica para o log se

Cumpre PELO MENOS 2 destes:

  • •Não trivial — afeta negócio, cliente, posicionamento, equipa, ou >2h/semana do operador
  • •Reversibilidade limitada — custa tempo/dinheiro/relações revertê-la
  • •Influencia futuras conversas com o Claude — o sistema deve manter coerência

Decisões operativas pequenas (cor, hora de reunião, nome de ficheiro) NÃO vão para o log. Para isso há tarefas e notas.

✓ O que FAZER

  • ✓Registar decisões quando se tomam (não dias depois)
  • ✓Incluir "Quando reconsiderar" concreto (data, métrica)
  • ✓Inverter decisão? NOVA entrada referenciando a anterior
  • ✓Decisões de cliente → log do cliente, pessoais → raiz
  • ✓Máximo 3 entradas por sessão (filtra ruído)

✗ O que NÃO fazer

  • ✗Apagar ou editar entradas (o log é histórico)
  • ✗Registar operações pequenas (cor, nome de ficheiro)
  • ✗"Quando reconsiderar: se correr mal" (vago)
  • ✗Registar decisões impulsivas sem reflexão
  • ✗Decisões contraditórias na mesma sessão sem contexto

💡 O paradoxo do log

As entradas mais úteis são as que doem registar — as que admites um erro, mudas de opinião, ou decides algo contra o teu instinto. Essas é que o log ajuda a respeitar.

🔬 /deep-dive — entrevista profunda

O wizard inicial foi o mínimo para arrancar. O /deep-dive aprofunda 22-25 dimensões que mudam muito os outputs. 25-30 min. Idempotente — retoma onde ficou.

Bloco Dimensões Ficheiro destino
A · Pessoa Horário produtivo, interrupções, contexto vital, motivadores, drenadores, estilo IA, palavras proibidas me.md + soul.md
B · Negócio Saúde financeira, margem, ticket médio, diferencial real, side projects, fricções do modelo work.md
C · Equipa Tamanho equipa (gate condicional), papéis, comunicação, delegação, clientes top, clientes problemáticos team.md
D · Foco Decisão pendente, meta 3 anos pro, meta 3 anos vital, medo profissional, métrica semanal, definição sucesso current-priorities.md + goals.md

📊 Anti-formulário

Não é checklist. É conversa adaptativa. As perguntas decidem-se em cada turno conforme a resposta anterior. Regras:

  • •Resposta curta → 1 follow-up (exemplo concreto, 5 whys leve, espelho)
  • •Máximo 2 níveis de profundidade por dimensão
  • •Branching: se trabalhas sozinho → salta bloco equipa (dimensões 15-17)
  • •Checkpoint a cada 7 dimensões: "Continuamos ou pausamos?"

💡 Pausável e retomável

Podes parar a meio: o sistema guarda deepDiveProgress e deepDiveLastDimension. Próxima vez /deep-dive retoma com abertura curta: "Da última vez ficámos em X. Faltam ~N dimensões. Continuamos?"

🩺 /doctor — diagnóstico + auto-fix

/doctor invoca a skill health-check v0.6 com validação profunda: JSON parseável, hooks executáveis, conteúdo >100 chars. Deteta state drift entre _install-state.json e o filesystem real.

10 passos do health-check

  1. Lê fonte de verdade — ~/.claude/skills/_install-state.json
  2. Verifica ambiente — Claude Code CLI, Node ≥18, Python 3, Git, .env válido
  3. Verifica Sinapsis profundo — JSON parseável, 5 hooks executáveis, ≥1 SKILL.md, hooks registados em settings.json
  4. Brand-context — voice-profile, samples, 3 registos, positioning, icp todos com >100 chars
  5. Context sectorizado — me, work, current-priorities, goals, team, decisions-log, learnings, soul (com drift check vs state)
  6. Skills curadas — 23 skills core presentes
  7. Settings — repo e global parseáveis, sem secrets hardcoded
  8. API keys — Firecrawl key formato correto
  9. Compila relatório 🟢🟡🔴 com state machine, drift, próximas ações
  10. Auto-fix limitado (com aprovação) — criar ficheiros vazios, chmod +x, re-registar hooks

⚠️ STATE DRIFT — caso especial

Se o state diz phase: done mas a validação profunda falha (ficheiro vazio, hook não executável), /doctor reporta STATE DRIFT.

Para reverter, requer confirmação literal "sí, revertir" (não basta "sim"). Reverte só o flag, NÃO apaga ficheiros. Depois /install --resume retoma a fase.

💡 Quando correr /doctor

Após update.sh · quando algo "não funciona" sem saber o quê · semanalmente como check preventivo · depois de fazer mudanças manuais em ficheiros do sistema · quando uma skill nova não ativa.

⬆️ update.sh — atualizar com conflict resolution

bash scripts/update.sh sincroniza o repo com o upstream do iAmasters OS. Nunca toca em user-data (clients/, brand-context/, context/, projects/, .env).

O que update.sh ATUALIZA O que update.sh NÃO toca
✅ Skills do repo (.claude/skills/) ❌ clients/<nome>/ (todos os teus clientes)
✅ Slash commands (.claude/commands/) ❌ brand-context/ (a tua marca)
✅ Templates de clientes (_templates/) ❌ context/ (me, work, decisions-log, learnings)
✅ Scripts (install, add-client, etc.) ❌ projects/ (os teus outputs)
✅ vendor/sinapsis/ (espelho upstream) ❌ .env (API keys)
✅ CLAUDE.md raiz + docs/ ❌ ~/.claude/ (Sinapsis state)

📊 Fluxo seguro

  1. git status — commit ou stash tudo o que tens pendente primeiro
  2. bash scripts/update.sh — faz backup automático antes
  3. Se houver conflitos em ficheiros que editaste, interativo: mantém o teu / aceita upstream / merge
  4. /doctor após update — confirma que tudo continua válido
  5. Se o state schema mudou (raro), /install --resume reaplica as fases necessárias

⚠️ Conflict resolution

Se editaste algum ficheiro do sistema (CLAUDE.md, uma skill core, um script), o update vai detetar conflito e perguntar:

  • •[1] Manter o meu — perde fixes do upstream, mas guarda customização
  • •[2] Aceitar upstream — perde a tua edição (backup no .backup/)
  • •[3] Merge manual — abre 3-way merge para resolveres

💡 Frequência

Update mensal está bem para a maioria. Se segues o release notes ativamente, semanal. Se nada do que sai te interessa, fica como estás — o OS funciona sem updates.
