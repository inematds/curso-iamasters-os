🤔 Quando usar (e quando não) multi-cliente

Multi-cliente não é "obrigatório". É uma capacidade opcional do OS. Ativá-lo sem o precisar adiciona fricção; não o ativar quando devias leva a copy errada, mistura de contextos e leaks de info entre clientes.

💡 Regra de ouro

Se trabalhas com 2+ clientes que têm voice, ICP e positioning distintos → multi-cliente. Caso contrário, fica na raiz.

O setup multi-cliente acrescenta ~30s por sessão (cd, claude, wrap-up). Compensa só quando há separação real para manter.

✓ Ativa multi-cliente se

  • ✓És freelance / agência com 2+ clientes em simultâneo
  • ✓Cada cliente tem brand voice distinta da tua
  • ✓Queres separação clara de outputs no filesystem
  • ✓Geras deliverables que não devem ter pegada tua
  • ✓Queres decisions-log independente por cliente

✗ NÃO ativar se

  • ✗És single-business (uma marca/produto só)
  • ✗Os "clientes" são verticais do mesmo produto
  • ✗Trabalhas só na tua marca pessoal
  • ✗Os clientes querem que uses a TUA voice (consultor cara visível)
  • ✗Estás a começar e tens 1 cliente piloto

🛠️ add-client.sh — criar cliente novo

Um comando, ~3 segundos, cliente pronto a trabalhar. O script add-client.sh valida o nome, clona um template, substitui placeholders e gera o CLAUDE.md de overrides.

Uso

bash scripts/add-client.sh <nome-cliente> [vertical]

# Exemplos
bash scripts/add-client.sh acme-corp freelance-ia
bash scripts/add-client.sh widget-shop agencia-marketing
bash scripts/add-client.sh north-star consultoria-b2b
bash scripts/add-client.sh nova-coach formador-online
bash scripts/add-client.sh xpto-startup vacio   # sem template

📊 O que o script faz (por baixo)

  1. Valida que o nome é kebab-case puro (regex ^[a-z0-9-]+$) — recusa "Acme Corp"
  2. Verifica que clients/<nome>/ NÃO existe (não sobrescreve)
  3. Cria estrutura: brand-context/{voice,positioning,icp,assets}, context/, projects/briefs/
  4. Clona ficheiros do template escolhido (voice-profile, positioning, icp, soul, user)
  5. Faz sed de {{CLIENT_NAME}} → nome real em todos os .md
  6. Gera CLAUDE.md do cliente com header, regras e referências ao brand-context
  7. Mostra a árvore criada e o próximo passo: cd clients/<nome> && claude

Estrutura resultante

clients/acme-corp/
├── CLAUDE.md                       # Overrides específicos
├── brand-context/
│   ├── voice/voice-profile.md      # Com placeholders do template
│   ├── positioning/positioning.md
│   ├── icp/icp.md
│   └── assets/
├── context/
│   ├── soul.md                     # Personalidade ajustada ao vertical
│   └── user.md                     # Template com campos típicos
└── projects/
    └── briefs/

💡 Dica prática

Mesmo com template, os ficheiros vêm com placeholders {{...}} que tens de preencher (cliente real, site, tom específico). O script avisa-te no fim com a lista exata.

📦 Os 4 templates verticais

Cada vertical é um ponto de partida com voice, positioning e ICP típicos do tipo de operação. Economiza 30-60 min de scaffolding por cliente novo.

👤 freelance-ia

Operador IA solo, 3-10 clientes

Voice: profissional mas acessível, formality 6/10, directness 8/10, warmth 7/10. Vocabulário proibido: "leverage", "transformação digital integral", "ecossistema holístico".

Inclui jargão técnico moderado (LLM, RAG, embeddings — se o cliente é técnico).

🏢 agencia-marketing

Equipa 2-10, MRR recorrente

Voice: criativa, pro, com toque humano. Diferente do freelance — fala como equipa ("nós", não "eu"). Vocabulário mais marketing-aware.

Cada cliente da agência tem A SUA voice — este template é da AGÊNCIA, não dos clientes dela.

🎓 formador-online

Cursos/comunidade, marca pessoal

Voice: didática, calorosa, com personalidade. Alta frequência. Tem que se sentir um humano específico, não "academia genérica".

Inclui padrões para LinkedIn, X, YouTube. Adapta-se ao registo da audiência.

💼 consultoria-b2b

High-ticket 30-300K€, formal

Voice: formal-autoridade, formality 8-9/10, humor 1-2/10. Decks, frameworks, propostas. Sem casualismos.

Vocabulário técnico-empresarial. Evita meme-talk, emojis, "uau".

📊 Como escolher

Se nenhum template encaixa bem, usa vacio e configura tudo de zero. Mas tenta primeiro o mais próximo — é mais rápido editar do que criar.

Os templates são ponto de partida, não destino. Edita o voice-profile depois da primeira semana real de trabalho com o cliente.

🧬 Herança CLAUDE.md (raiz + cliente)

Quando fazes cd clients/acme && claude, o Claude lê dois CLAUDE.md em cascata: o da raiz (instruções gerais) e o do cliente (overrides específicos). Saber a ordem evita conflitos.

1

CLAUDE.md raiz (do repo)

Regras gerais que se aplicam a TODOS os clientes + ao operador

Idioma, convenções de pastas, install gate, paths, skills registry. Não tem regras específicas de cliente — só sistema.

2

clients/<X>/CLAUDE.md (do cliente)

Overrides específicos — só ativos quando estás cd no cliente

Nome do cliente, template base, regras particulares ("nunca usar 'vocês' formal"), skills prioritárias, notas operativas, referências ao brand-context do cliente.

3

brand-context/ + context/ do cliente

Carregados automaticamente pelas skills marketing-*

Voice profile, ICP, positioning, soul, user — todos do cliente. Skills da raiz (.claude/skills/) continuam disponíveis e operam sobre o brand-context do cliente.

💡 Skills custom por cliente

Se um cliente requer skill que NÃO se aplica a outros, cria em clients/<X>/.claude/skills/<categoria>/<nome>/.

Essa skill só está disponível quando estás cd nesse cliente. Não polui os outros nem a raiz.

⚠️ Sinapsis é global

O operator-state.json de Sinapsis é UM, em ~/.claude/skills/, partilhado entre todos os clientes. Aplica-se a ti como operador, não ao cliente. É o esperado — tu és o mesmo profissional para todos.

🔄 Workflow diário multi-cliente

Um ritual simples que respeita a separação. Sempre cd antes de claude. Sempre /wrap-up antes de mudar de cliente.

Segunda-feira típica · 3 clientes

# Cliente A (freelance-ia)
cd clients/acme-corp && claude
> "Cria post LinkedIn sobre case study X"
  → marketing-copywriting gera com voice de Acme
> /wrap-up
exit

# Cliente B (agencia-marketing)
cd ../widget-shop && claude
> "Repurpose último vídeo do CEO em 5 peças"
  → marketing-content-repurposing
> /wrap-up
exit

# Cliente C (consultoria-b2b)
cd ../north-star-consult && claude
> "Redige proposta comercial para novo lead"
  → marketing-copywriting com registo A formal
> /wrap-up
exit

✓ Best practices

  • ✓Sempre cd antes de claude
  • ✓/wrap-up ao mudar de contexto (daily summary separa-se por cliente)
  • ✓Brand voice por cliente, não partilhar
  • ✓2 terminais para sessões simultâneas em clientes diferentes
  • ✓Backup separado para clientes com info crítica

✗ Anti-patterns

  • ✗Trabalhar conteúdo de cliente desde a raiz do repo
  • ✗Mudar de cliente sem wrap-up (perdes summary)
  • ✗1 sessão a saltar entre clientes via cd interno
  • ✗Copiar voice profile de um cliente para outro
  • ✗Esquecer-te de qual cliente estás (verifica com pwd)

📊 Operador próprio + clientes

Quando trabalhas na tua marca pessoal (LinkedIn próprio, blog, marca operador): fica na raiz. Skills marketing-* usam o teu brand-context da raiz.

Quando trabalhas com/para um cliente: cd clients/<X>. Skills usam o brand-context do cliente. Regra: nunca geres conteúdo de cliente desde a raiz.

6

🔒 Segurança: separação de info entre clientes

Por design, o OS não partilha info entre clientes automaticamente. Mas a disciplina diária é tua — o sistema apenas remove a maioria das oportunidades de erro.

Mecanismo O que protege Limitação
.gitignore de clients/ Não exporta clientes para git público Tens que mover para repo privado se quiseres sync
Skills isoladas por cd Skills marketing-* só veem o cliente atual Falha se ignoras a regra do cd
update.sh skip user-data Updates do OS nunca tocam em clients/<X>/ Templates de _templates/ podem mudar — copy é manual
Brand-context separado Voice/ICP de A não influencia copy de B Operator-state (Sinapsis) é global — tu és o mesmo
Decisions-log por cliente Decisões de A não aparecem em sessões de B Decisões operador-próprio ficam na raiz

💡 Reutilizar info entre clientes

Se queres usar aprendizagens do cliente A para o cliente B:

  1. Faz manual e consciente (não acidental)
  2. Gera "case study anonimizado" como output do cliente A
  3. Referencia esse case study no cliente B

⚠️ Troubleshooting comum

  • "O Claude não aplica voice do cliente" → verifica pwd (estás no cliente?), voice-profile.md está preenchido (sem {{...}}), CLAUDE.md do cliente referencia o path correto.
  • "Skills da raiz não aparecem" → reinicia Claude Code (Ctrl+C × 2 + claude).
  • "Misturo info acidentalmente" → sinal de não respeitar cd. Implementa: sair do Claude entre clientes, NÃO abrir 2 sessões na mesma terminal.
