📥 /install-skill — instalar skill externa de GitHub

/install-skill <github-url> traz uma skill de outro repo para o teu, mas com validação prévia que bloqueia skills mal feitas. Anti-inflação por design.

7 passos do comando

  1. Valida URL — é GitHub? aponta para pasta ou SKILL.md?
  2. Descarrega temporário — scripts/validate-skill.sh usa git archive para subpasta
  3. Valida estrutura — obrigatório (SKILL.md, YAML, description 50-500 chars, verbo de intenção), recomendado (references/ se >2500 chars, examples.md, sem eval()), bloqueante (sem SKILL.md, YAML mal formado, scripts destrutivos)
  4. Mostra resultado ao operador — conflitos com locais, tamanho, validações
  5. Instala localmente — backup automático se conflito, copia para .claude/skills/<categoria>/<nome>/, atualiza catalog e registry
  6. Teste de ativação — reiniciar Claude, prompt que deveria ativar, oferece refinar description se falhar
  7. Fecho — sugere /wrap-up, append em learnings.md se trouxe valor

Exemplo de validação

## Validação de skill

URL: https://github.com/scrapes/skills/humanizer
Skill: tool-humanizer (atenção: já tens uma skill local com esse nome!)
Tamanho: SKILL.md 2.1KB, references 4.5KB, scripts 0KB

### Validações
✅ SKILL.md presente
✅ YAML frontmatter correto
✅ description 187 chars (no intervalo)
⚠️ Conflito: já existe .claude/skills/tools/tool-humanizer/
✅ Sem código executável perigoso
✅ Sem credenciais hardcoded

### Ação
[1] Cancelar
[2] Substituir a tua local (com backup automático)
[3] Instalar como tool-humanizer-v2 (renomear)
[4] Ver diff vs a tua versão local

💡 Modo atalho

Se o argumento NÃO é URL mas nome simples (ex. /install-skill cognito), o comando procura em .claude/skills/_meta/_optional/ e move para ativa. Útil para skills opcionais já incluídas no repo.

🔌 /install-mcp — instalar MCP server

MCPs (Model Context Protocol servers) ligam o Claude ao teu stack real: BD, CRM, repos, docs. /install-mcp <name> instala desde lista curada ou URL custom.

MCP Para que Plano
context7 Docs vivos de frameworks (Next.js, Supabase, Tailwind) injetados no contexto. Evita alucinação de APIs obsoletas. Grátis
github Ler issues/PRs/ficheiros, criar/comentar issues, mergear PRs sem sair do Claude. Grátis (PAT)
supabase Queries SQL, gestão tabelas, RLS, edge functions. READ-ONLY token para começar. Grátis (OSS)
notion Ler páginas, criar BDs, mover páginas. Se o teu wiki/CRM está no Notion. Grátis (limite)
firecrawl Scraping de sites com bot blockers. Usado por skills marketing-*. NÃO vai em .mcp.json, vai em .env. 500 free + Hobby $16/mo

Estrutura típica de .mcp.json

{
  "mcpServers": {
    "context7": {
      "command": "npx",
      "args": ["-y", "@upstash/context7-mcp"],
      "env": {}
    },
    "supabase": {
      "command": "npx",
      "args": ["-y", "@supabase/mcp-server-supabase@latest",
               "--project-ref=$SUPABASE_PROJECT_REF"],
      "env": {
        "SUPABASE_ACCESS_TOKEN": "$SUPABASE_ACCESS_TOKEN"
      }
    }
  }
}

⚠️ Token budget

Cada MCP ativo adiciona contexto ao system prompt (descrições das suas tools). 5-7 MCPs ativos máximo. Se precisas mais, comenta os que não usas frequentemente, ou mantém .mcp.json diferente por cliente.

READ-ONLY antes de write em todos os MCPs com permissões sensíveis (BD, email, mensagens). Eleva permissões só quando confiares no fluxo.

💡 MCPs a evitar em produção

Auto-post a redes sociais (LinkedIn, X, IG) sem gates — risco de fuga de identidade. MCPs sem documentação clara de scopes. MCPs custom de devs desconhecidos sem auditar o código.

🔧 mcps-curated.md

MCPs validados para operadores IA

  • • ⭐ Top 5: context7, github, supabase, notion, firecrawl
  • • Úteis específicos: linear, gmail (read-only), slack, filesystem (whitelisted)
  • • A evitar: auto-post redes sociais, MCPs sem scopes claros, custom não auditados

📊 Por que catálogo curado vs marketplace livre

  • •Filtro de qualidade: cada entrada testada ≥2 semanas em produção, não "instalei e parece OK"
  • •Anti-inflação cognitiva: 1000 skills paralisa. 30 bem escolhidas habilitam
  • •Revisão trimestral: skills obsoletas saem, skills novas validadas entram
  • •Curador identificado: maintainer (Angel) revê PRs — não anonimato

💡 Alternativa lean: claude-code-second-brain

Se preferes minimal sem skills curadas, claude-code-second-brain de Luis Pitik (autor de Sinapsis) é a versão lean: prompt → entrevista → ficheiros context/ + decisions-log. São produtos irmãos, não concorrência.

📄 Plugins Anthropic via marketplace

Para ler/criar/editar ficheiros Office (Word, Excel, PowerPoint, PDF) sem libs externas, instala o plugin anthropic-skills — 4 skills oficiais Anthropic com manutenção garantida.

Instalação

# Dentro de Claude Code
/plugin install anthropic-skills

# Ativa 4 skills:
#   docx — Word documents
#   xlsx — Excel/CSV/TSV spreadsheets
#   pptx — PowerPoint presentations
#   pdf  — Read, combine, split, mark PDFs

📝 docx

Criar, ler, editar documentos Word. Útil para propostas, contratos, briefings que vêm/vão em .docx (a maioria dos clientes corporativos).

📊 xlsx

Manipular spreadsheets (xlsx, xlsm, csv, tsv). Reports mensais, financeiros, listas de clientes — tudo em Excel.

🎯 pptx

Criar e editar apresentações. Decks comerciais, decks de status, training slides — geras do prompt em vez de slide-a-slide manual.

📃 pdf

Ler, combinar, dividir, marcar PDFs. Auditar contratos, extrair info de PDFs de clientes, montar dossiers.

✓ Recomendação: instalar SEMPRE

Para qualquer operador que receba ou entregue Office (a maioria dos clientes B2B). Skills oficiais Anthropic = manutenção garantida + segurança auditada.

📊 Marketplace Anthropic vs install-skill (GitHub)

  • •Marketplace (/plugin install): skills oficiais Anthropic ou parceiros, auditadas, manutenção contínua
  • •install-skill GitHub (/install-skill <url>): comunidade, requer validação local antes de instalar
  • •Para Office, marketplace SEMPRE. Para skills de marketing/CRO, catálogo iAmasters via install-skill

✍️ Criar a tua própria skill com meta-skill-creator

Quando detetas um padrão repetido em 3+ sessões, é hora de o "graduar" para skill. A skill meta-skill-creator faz scaffolding completo seguindo o padrão canónico iAmasters.

Contrato de qualidade de uma skill iAmasters

  1. YAML frontmatter completo — name kebab-case com prefixo, description 50-500 chars com quando ativar
  2. Progressive disclosure — SKILL.md não tem tudo; references/ guarda knowledge extenso
  3. Passos numerados e testáveis — cada passo verificável
  4. Skill collaboration explícita — se invoca outras skills, nomeia quando e porquê
  5. Output verifier gate se gera conteúdo entregável
  6. Learnings hook — no fim regista em context/learnings.md
  7. Idioma — SKILL.md em português, code/JSON em inglês

Estrutura de pastas gerada

.claude/skills/<categoria>/<nome>/
├── SKILL.md                    # Processo principal
├── references/                 # (opcional) Knowledge separado
│   ├── examples.md             # 2-3 exemplos de uso real
│   ├── checklist.md            # (opcional) Validações QA
│   └── <outros>.md
└── scripts/                    # (opcional) Se requer executáveis
    └── <nome>.py               # ou .sh

📊 3 testes da descrição

  • •Ativação: um Claude que só lê a descrição saberia quando usar? Contém verbos de intenção (cria, gera, analisa, extrai)?
  • •Comprimento: 50-500 chars. Menos é ambíguo, mais é inflado
  • •Unicidade: lê skills-catalog.json — outra skill com descrição parecida causa canibalização (Claude não saberá qual escolher)

⚠️ Teste de ativação obrigatório

Antes de declarar a skill terminada:

  1. Fechar Claude Code (Ctrl+C × 2) e reabrir
  2. Prompt que deveria ativar a skill segundo a description
  3. Verifica que Claude a escolhe e segue passos sem saltar fases
  4. Se não ativa → refinar a descrição. Se ativa mas falha → refinar os passos

💡 Edge cases comuns

  • • Skill demasiado genérica ("uma skill que escreva bem") → pede concretude: para que plataforma? que tom? output onde?
  • • Já existe skill parecida → mostrar ambas, oferece ampliar/diferenciar/cancelar
  • • Skill demasiado pequena (1 passo) → pode ser melhor slash command em .claude/commands/
  • • Sem categoria óbvia → cria categoria nova só se vai ter 3+ skills

🤝 Contribuir ao catálogo

As skills mais úteis que criares para ti podem servir 100 outros operadores. O catálogo iAmasters aceita PRs com critérios objetivos e revisão trimestral.

Critérios para uma skill entrar no catálogo

  • ✓Serve a ≥3 avatares iAmasters (freelance-ia, agencia-marketing, formador-online, consultoria-b2b)
  • ✓Output útil sem contexto prévio do operador (não depende do teu setup específico)
  • ✓Não depende de integrações privadas (Skool, Wixin, ferramentas fechadas)
  • ✓Não depende de informação confidencial (clientes, decisões internas)
  • ✓Validada em produção ≥2 semanas — não "instalei e parece OK"

📊 Processo de PR

  1. Abre issue ou PR em iamasters-academy/iamasters-os
  2. Inclui: link à skill, 1-2 exemplos de uso real, categoria sugerida, porque encaixa na Camada 2
  3. Maintainer (Angel) revê — pode pedir ajustes (description, examples, edge cases)
  4. As que passam adicionam-se ao catálogo na próxima release trimestral
  5. Skills obsoletas saem na mesma revisão — catálogo vive, não cresce indefinidamente

✓ Devolver é a melhor forma de aprender

Escrever uma skill para "ti" é diferente de a escrever para "outros operadores". Quando submetes, és forçado a generalizar, documentar bem, pensar em edge cases que não tinhas no teu setup. O catálogo melhora-te a ti, não só ao ecossistema.

💡 Antes de PR, instala-a "como externa"

Publica a tua skill no teu GitHub público. Pede a 2-3 colegas para a instalarem com /install-skill <url>. Vê se a description ativa nos casos certos, se os passos correm, se a documentação chega. Esse feedback vale mais que o PR direto.
