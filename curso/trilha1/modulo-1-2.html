📋 Prerequisitos do sistema

Antes de correres install.sh precisas de 4 dependências críticas no teu sistema. O instalador valida-as e bloqueia se faltar algo essencial — não vais ficar com meio-OS instalado por erro de Node em falta.

💡 Conceito principal

O install.sh tem uma fase prereqs dedicada que faz deteção multi-plataforma. Não basta "tens python"; tem de ser Python 3.9-3.13, com binário correto (não o stub da Microsoft Store), encontrável em pelo menos um dos comandos python3, py -3, python, python3.11, etc.

  • •Git ≥2.0 — para clonar o repo e atualizações futuras
  • •Node.js ≥18 — obrigatório, os hooks Sinapsis usam-no
  • •Python ≥3.9 — opcional mas recomendado (hooks de observação)
  • •Claude Code CLI ou Desktop — onde o OS vai correr

Output esperado do install.sh na fase prereqs

── [1/2] A validar prerequisitos ──
[OK]    OS: linux (x86_64)
[OK]    git: 2.43.0
[OK]    node: v20.10.0
[OK]    python: python3 (Python 3.11.7)
[OK]    Claude CLI no PATH

✓ Instalações OK

  • ✓macOS com Homebrew (brew install node python git)
  • ✓Linux Ubuntu/Debian via apt
  • ✓Windows + WSL2 (Ubuntu) — experiência mais limpa
  • ✓Python desde python.org (Windows)

✗ Problemas frequentes

  • ✗Windows + Python da Microsoft Store (stub, não executa)
  • ✗Node v16 ou inferior — Sinapsis precisa de v18+
  • ✗Git Bash sem WSL — funciona mas com latência maior
  • ✗nvm a apontar para versão antiga numa shell

💡 Dica prática

Em Windows, se Python não é detetado mas tens a certeza que está instalado, abre PowerShell e corre where.exe python. Se aparece um path em WindowsApps/, é o stub da Store — desinstala e usa o instalador oficial de python.org marcando "Add to PATH".

2

🔄 Install.sh em 6 passos

O scripts/install.sh é idempotente e reentrante. Podes correr 10 vezes — só refaz o que falta. Está estruturado em 6 fases técnicas (3 do script + 3 dentro do Claude Code).

1

Validar prerequisitos

Deteção de OS, git, node, python, claude. Falha bloqueante se Node <18 ou Git ausente. Resultado registado em phases.prereqs.checks.

2

Instalar Sinapsis engine

Executa vendor/sinapsis/install.sh que copia ~/.claude/ com 23 skills + hooks + operator-state. Validação profunda pós-instalação: JSON parseable, hooks com +x, ≥1 SKILL.md.

3

Configurar camada OS

Cria projects/, context/, brand-context/. Gera soul.md, decisions-log.md, learnings.md com headers canónicos. Copia .env.example → .env.

4

Instalar Install Gate hook

Copia _install-gate.sh para ~/.claude/skills/ com chmod +x. Regista-o como hook SessionStart em ~/.claude/settings.json (idempotente — não duplica).

5

Wizard conversacional (no Claude Code)

Já não é o bash — abres Claude Code e o hook deteta fases pendentes. A skill meta-onboarding-wizard faz 4 sub-fases (identidade, negócio, prioridades, objetivos) e escreve context/me.md, work.md, etc.

6

Welcome quick-win

A skill welcome-quick-win gera o primeiro deliverable em projects/welcome/ usando o brand context recém-criado. Sinal claro de que o sistema funciona end-to-end.

📊 Output estruturado [parseable]

Todas as linhas do instalador seguem o formato [CATEGORIA] mensagem:

  • [OK] — fase completada com sucesso
  • [SKIP] — já existia / não se aplica · motivo incluído
  • [WARN] — continua mas com limitação
  • [ERROR] — bloqueante, fase fica failed no state
  • >> — info contextual (não muda state)

Este formato permite grep '\[ERROR\]' para resumo rápido se algo falhar.

💡 Dica prática · flags úteis

  • bash scripts/install.sh — execução normal · idempotente
  • bash scripts/install.sh --resume — continua desde última fase não-done
  • bash scripts/install.sh --force-reinstall — backup do state e arranque fresco (não apaga ficheiros pessoais)
  • bash scripts/install.sh --skip-sinapsis — debug, salta Sinapsis (raro)
3

🗄️ State machine em _install-state.json

Este é o coração do anti "instalação fantasma". Um ficheiro JSON em ~/.claude/skills/_install-state.json que regista o estado real de cada fase. Não é uma opinião do agente — é o contrato entre o instalador bash, as skills do wizard, e o gate.

💡 Conceito principal

Antes (v0.5.x) o script bash era um único disparo + wizard. Se algo falhasse a meio, o agente podia reportar "tudo bem" sem que fosse verdade. Desde v0.6: cada fase tem estado persistente e verificável independentemente do que o agente diga.

  • •Source of truth única e externa ao modelo
  • •Resume automático onde ficou (reentrante)
  • •Validação profunda pós-cada-fase (não confia em declarações)

📊 Leitura

Operador começou instalação, terminou prereqs e Sinapsis, abriu Claude Code, o wizard arrancou context-files, escreveu me.md, e o operador disse "para, volto depois". Na próxima sessão, o hook lê isto e força /install --resume que retoma desde work.md. Nada se perde.

💡 Dica prática

Podes inspecionar o state a qualquer momento com cat ~/.claude/skills/_install-state.json | jq. Útil para perceber porque o agente diz "instala primeiro" quando achas que já instalaste. O ficheiro não mente — se diz pending, é porque está pending.

4

⛔ Install Gate · o hook SessionStart

O _install-gate.sh é um hook bash + node que corre antes do modelo ver a tua primeira mensagem em cada sessão. Lê o state machine e, se há fases pendentes, injeta um additionalContext que instrui o agente a NÃO responder ao teu pedido até cerrar a instalação.

💡 Conceito principal

É a peça que distingue v0.6 de versões anteriores: enforcement estrutural, não confiança no modelo. O hook não pede ao modelo "por favor verifica se está instalado". Injeta a verdade do disco antes do modelo poder responder.

Resultado: torna-se quase impossível "instalações fantasma" onde o modelo declara sucesso sem que os ficheiros existam.

Comportamento por estado

Caso 1 · State file NÃO existe

Se ~/.claude/skills/_install-state.json não existe E estás num repo iamasters-os (validado por vendor/sinapsis/ + CLAUDE.md), injeta:

[INSTALL GATE] iAmasters OS detetado mas NÃO instalado.
Indica ao utilizador: 'Preciso que executes: bash scripts/install.sh'
NÃO crias ficheiros manualmente para simular instalação.
Caso 2 · State existe, fases pendentes

Calcula required total e done, lista failed/in-progress/pending, e injeta instruções numeradas de 1 a 7 que o modelo deve seguir antes de responder. Inclui ações específicas por fase pendente (se é context-files, invoca o wizard; se é welcome-completed, invoca welcome-quick-win).

Caso 3 · Tudo done

Sai com exit 0 sem output. O agente nunca vê nada — passa direto para a tua mensagem.

Caso 4 · State corrompido

Injeta: "State exists but is corrupted. Tell user to run: bash scripts/install.sh --force-reinstall". Nunca crasha silencioso.

Anatomia do hook (excerpt)

#!/bin/bash
set +e  # Nunca queremos que um hook partido bloqueie a sessão

STATE_FILE="$HOME/.claude/skills/_install-state.json"

# Se não temos node, saímos silencioso (fallback)
if ! command -v node >/dev/null 2>&1; then exit 0; fi

# Caso 1: state file NÃO existe → OS não instalado
if [ ! -f "$STATE_FILE" ]; then
    if [ -d "$PWD/vendor/sinapsis" ] && grep -q "iAmasters OS" "$PWD/CLAUDE.md"; then
        cat <<'EOF'
{"hookSpecificOutput":{"hookEventName":"SessionStart",
 "additionalContext":"[INSTALL GATE] iAmasters OS detetado mas NÃO instalado..."}}
EOF
    fi
    exit 0
fi

# Caso 2: state existe → avaliamos
node << NODE_EOF
const s = JSON.parse(fs.readFileSync('$STATE_FILE','utf8'));
const required = Object.entries(s.phases).filter(([k,v]) => v.required === true);
const pending = required.filter(([k,v]) => v.status !== 'done');
if (pending.length === 0) process.exit(0);  // Tudo OK, sair silencioso
// Senão, construir mensagem detalhada com pending/failed/in-progress
// e fazer process.stdout.write(JSON.stringify({hookSpecificOutput: {...}}))
NODE_EOF

⚠️ Fallback se node não existe

O hook detecta ausência de Node e sai silencioso (exit 0). Racional: se Node não está, o Sinapsis também não funciona — o OS está partido na mesma. Não vale a pena bloquear a sessão por causa do hook quando o sistema em si não vai funcionar.

5

📊 Comandos /install e /install-status

Estes dois slash commands são a única via oficial para gerir a instalação a partir de dentro do Claude Code. Um orquestra, outro reporta.

/install — orquestrador

Lê state, identifica próxima fase, executa-a (ou diz-te para correres bash no terminal se for fase técnica).

  • •Reentrante — corres as vezes que for preciso
  • •--resume (default) retoma desde a última pendente
  • •--force-reinstall requer confirmação explícita
  • •Sabe distinguir fases técnicas (bash) vs conversacionais (Claude Code)

/install-status — dashboard

Read-only. Mostra o estado sem tocar em nada. Útil quando duvidas se algo está bem.

  • •NÃO modifica state
  • •NÃO executa fases
  • •Mostra: lista de fases, último erro, próximo passo sugerido
  • •Seguro de correr a qualquer momento

Fluxo decisional do /install

A
Tudo done

Diz "já está tudo instalado, não há nada a fazer" e aborta. Sugere /doctor se algo te parecer estranho.

B
Fase técnica pendente (prereqs ou sinapsis-engine)

Diz "executa bash scripts/install.sh --resume no terminal" — não pode correr bash com privilégios suficientes a partir do chat.

C
Fase conversacional pendente

Invoca diretamente: context-files/operator-state → meta-onboarding-wizard. welcome-completed → welcome-quick-win. deep-dive-completed → meta-deep-dive.

D
Fase failed

Mostra o último erro de errors[], propõe fix concreto, e oferece-se para retentar.

💡 Dica prática

Se o agente disser "tens a instalação completa" mas tens dúvidas, corre /install-status. É 100% confiável porque lê o ficheiro JSON diretamente. Se o status mostra done em todas as required mas algo continua estranho, então é caso para /doctor (drift detection).

6

🩺 /doctor e drift detection

A skill health-check (invocada por /doctor) é a tua segunda linha de defesa. O state pode dizer que está tudo done, mas tu podes ter apagado um ficheiro por engano, mudado uma config, ou um update partiu algo. /doctor faz validação profunda e compara com o state para detetar drift.

💡 O que é drift

Drift = desalinhamento entre estado declarado e estado real. Dois casos típicos:

  • •State diz done, ficheiros faltam — apagaste context/me.md; /doctor deteta e propõe reverter state para in-progress para o wizard reativar
  • •State diz pending, ficheiros existem — atualizaste manualmente; /doctor pode re-validar e marcar done

10 passos do health-check

Passo 0

Lê _install-state.json como source of truth. Se não existe → 🔴 instalação não iniciada.

Passo 1

Verificação de ambiente · Node ≥18, Python 3, Git, Claude Code detetado, .env sem sintaxe partida.

Passo 2

Sinapsis profundo · JSONs parseáveis, 5 hooks +x, settings.json com secção hooks.

Passo 3

Brand context · voice-profile.md, samples, registers A/B/C, positioning, ICP >100 chars.

Passo 4

Context sectorizado + drift check · 4 críticos com nome real / negócio / prioridades / objetivos.

Passo 5

23 skills core presentes. Faltantes → 🟡 sugere bash scripts/update.sh.

Passo 6

Settings JSON válido (repo + global), sem permissões perigosas, sem secrets hardcoded.

Passo 7

API keys opcionais · FIRECRAWL_API_KEY em .env com formato correto.

Passo 8

Compilar relatório 🟢🟡🔴 estruturado por categoria com contadores agregados.

Passo 9

Auto-fix limitado · só problemas reversíveis e seguros, com confirmação.

🚨 Fix de DRIFT requer confirmação explícita extra

Quando /doctor deteta drift, NÃO mexe sozinho. Pede confirmação literal:

🚨 STATE DRIFT detetado na fase "context-files".
O state diz "done" mas a validação profunda falha.

Quero reverter o state para "in-progress" para que o sistema
force re-executar a fase. Isto NÃO apaga ficheiros — só muda o flag.

Confirmas? (escreve "sim, reverter")

Se respondes qualquer coisa diferente de "sim, reverter", NÃO toca em nada. Segurança first.

💡 Dica prática · quando correr /doctor

  • Após cada update.sh — para confirmar que a atualização não partiu nada
  • Quando algo "soa estranho" sem saberes o quê — output esquisito, skill que não ativa
  • Antes de criar cliente novo — para evitar arrancar um cliente sobre um sistema partido
  • 1 vez por mês como rotina sanitária — apanha drift acumulado
